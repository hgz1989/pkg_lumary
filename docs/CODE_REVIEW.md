# 项目代码审查与优化建议清单

本次审查主要针对架构一致性、运行时潜在风险、代码风格及可维护性，整理出以下问题及优化建议。

## 1. 确定性风险与问题 (高优先级)

| No. | 发现的问题 | 具体表现与风险 | 优化建议 | 涉及文件 |
|---|---|---|---|---|
| **1.1** | **顶层导出失效** | 顶层 `__all__` 导出了 `BaseService`，但文件内未实际导入，导致 `from lumary import BaseService` 报错。 | 补齐 `from .services import BaseService` 导入，或从 `__all__` 中移除。 | `lumary/__init__.py` |
| **1.2** | **README 示例失真** | README 中实例化 `Lumary` 时使用了 `enable_health_check=True`，但实际构造函数无此参数（健康检查为默认注册）。 | 修正 README 文档示例，或在构造函数中补充该参数。 | `README.md`, `lumary/application.py` |
| **1.3** | **子应用生命周期冲突** | 子应用初始化时主动移除了 `root_path`，但生命周期钩子又强制要求 `root_path` 非空才允许启动，这会导致独立运行子应用时发生自相矛盾。 | 重新梳理“子应用独立运行”与“作为子应用挂载”两种模式的生命周期校验逻辑。 | `lumary/application.py` |
| **1.4** | **业务异常协议风险** | `BusinessException` 及其所有子类（含 401/403/404）均被强制返回 HTTP 200，仅通过响应体错误码区分。 | 明确这是否为强约定。若是，需在文档中显著声明；否则建议恢复 HTTP 语义状态码，以免误导网关和缓存层。 | `lumary/handlers.py`, `lumary/exceptions.py` |
| **1.5** | **全局异常无堆栈** | 兜底的全局异常 `global_exception_handler` 仅记录了异常消息（`logger.critical`），未记录 traceback 堆栈信息。 | 添加 `exc_info=True` 或改用 `logger.exception`，以便线上排障。 | `lumary/handlers.py` |
| **1.6** | **默认 CORS 兼容风险** | 默认 CORS 配置同时使用了 `allow_origins=['*']` 与 `allow_credentials=True`，在浏览器凭证跨域场景下有规范兼容风险。 | 收紧默认策略，或在文档中明确这是仅用于开发的配置。 | `lumary/middleware.py` |
| **1.7** | **测试覆盖严重不足** | `tests/` 目录下仅有启动脚本，无任何真实的测试用例，关键的异常处理、子应用挂载等核心逻辑毫无测试保障。 | 增加基础的 pytest 测试用例，优先覆盖核心逻辑和 README 中的用法。 | `tests/test_lumary.py` |

## 2. 风格与可维护性 (中/低优先级)

### 2.1 模块头与 Docstring
- **表现**：大多模块开头使用“作者/日期”模板，但 `@Description` 为空；部分类和函数的 docstring 包含多余的实现细节，且 `Returns` 写法与 Google 风格要求不一致。
- **建议**：回归现代 Python PEP 257 风格，让模块级注释真正说明“模块的职责和功能”，精简方法级注释。

### 2.2 注释过度符号化
- **表现**：注释密度偏高，大量使用分隔线，且混入较多符号/Emoji（如 `👇, ✅, 🔥, ❌`）。
- **建议**：在基础库和框架代码中，尽量保持沉稳、简洁的注释风格，多依靠命名和代码结构本身来表达逻辑。

### 2.3 导入与包暴露边界
- **表现**：`lumary/__init__.py` 暴露了大量 FastAPI/Pydantic 的原生对象，导致 `lumary` 变成了这些外部库的“二次命名空间”，模糊了框架自有能力的入口。
- **建议**：收敛顶层导出，重点暴露 `Lumary`、生命周期、业务异常、WS 管理器等自有核心组件，让用户清楚“什么是框架提供的”。

### 2.4 版本号多点硬编码
- **表现**：版本号 `0.1.6` 同时硬编码在 `pyproject.toml`, `lumary/__init__.py`, `lumary/application.py` 中。
- **建议**：统一版本号来源（如通过 `importlib.metadata` 动态获取，或集中在一个 `__version__.py` 文件中读取），防止未来更新时出现版本漂移。

### 2.5 `apps` 目录定位不清
- **表现**：`apps` 目录在打包配置中被排除，但核心代码又强依赖它的目录约定进行子应用动态挂载；导致“框架代码”和“示例应用”边界模糊。
- **建议**：明确 `apps` 的定位。如果它是官方推荐的插件体系，应补充完整文档和规范；如果仅为示例，建议移至 `examples/` 目录。
